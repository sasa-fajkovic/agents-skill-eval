How it works

Three steps from upload to structured feedback.

The flow is intentionally simple and inspection-friendly. Every stage is designed to make the evaluation understandable rather than opaque.

01 Upload `SKILL.md` plus supporting files

Provide a local folder or GitHub reference so the evaluator can inspect the full skill package instead of a disconnected markdown file.

02 Run deterministic checks in a sandbox

The package is validated and evaluated inside an isolated container. Optional Anthropic review runs only after that deterministic step.

03 Get structured reasons for every flag

Results include score, quality tier, deterministic checklist, and optional review fields that are easy to scan and compare.

FAQ

Common design questions, answered directly.

The short version: skills are a portable capability format, runtime-specific slash commands are not, and portable helper scripts matter if a skill should survive across environments.

Why agent skills over slash commands?

Skills carry trigger conditions, inputs, outputs, examples, and failure handling. Slash commands are useful shortcuts, but they are not a durable capability contract.

Read the full answer
Why evaluate the IO spec instead of Claude Code specifics?

The evaluator targets portable skill definitions so one runtime's proprietary ergonomics do not dominate the score.

Read the full answer
Why prefer `.sh` and `.py` helper scripts?

They are the most portable options across Linux and macOS environments. Runtime-heavy script formats make skills harder to reuse safely.

Read the full answer
Open source and self-hostable?

The project is public and can be deployed without a frontend build pipeline. The evaluator is designed to ship as raw HTML, CSS, JS, and a Go backend.
