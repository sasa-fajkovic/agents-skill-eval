Positioning

Why agent skills over slash commands?

Slash commands are useful shortcuts, but they are a weak foundation for portable, evaluable agent capabilities. This platform focuses on skills because skills can be activated by context, inspected systematically, and reused across runtimes.

Why not slash commands?
  • Commands are imperative, one-shot triggers. They do not naturally carry activation context, state, or structured IO contracts.
  • Commands are usually tightly coupled to one runtime such as Claude Code `/commands`, Codex prompts, or another vendor-specific invocation model.
  • Commands rarely define when they should activate, what they require, or what they return. They depend on the user already knowing how and when to use them.
  • That makes them harder to evaluate consistently. A command may be convenient, but it is not the same thing as a portable capability definition.
Why SKILL.md?
  • Skills are declarative. A good skill defines trigger conditions, required inputs, expected outputs, and behavior in a structured, readable format.
  • Skills are self-documenting and evaluable. You can review their structure, activation logic, examples, edge cases, and coverage.
  • Skills can activate autonomously when the surrounding context matches, instead of relying on manual invocation every time.
  • Skills are where the ecosystem is heading. Anthropic, VoltAgent, Composio, and related tools are converging on reusable skill and capability definitions rather than one-off runtime commands.
Agent-Agnostic Evaluation

Why evaluate the IO spec instead of a specific agent?

agents-skill-eval.com focuses on the portable skill definition itself: the `SKILL.md` structure, trigger definitions, and input/output contract. It does not score a skill based on Claude Code-only or Codex-only conventions.

Portable by design
  • A well-written skill should be portable. It should work across Claude Code, Codex, Gemini CLI, OpenClaw, Cursor, and any other agent runtime that implements the skill standard.
  • Evaluations focus on portability, clarity, and reliability of the skill definition itself, not whether it uses one runtime's proprietary syntax or ergonomics.
  • The goal is vendor-neutral, runtime-agnostic agent capabilities. Locking skills to proprietary runtime features weakens portability and makes skills harder to reuse and compare.
  • That is why the platform evaluates the IO specification and behavior contract first. The runtime should be interchangeable; the skill definition should remain understandable and useful across environments.
Script Guidance

Preferred script types inside skills: `.sh` and `.py`

Bundled skill scripts should favor shell (`.sh`) and Python (`.py`) because they are the most portable options across Linux and macOS environments where agent skills are likely to run.

Why `.sh` and `.py`?
  • Both are commonly available on Linux and macOS without requiring an extra runtime manager.
  • Shell scripts run directly through `/bin/sh` or `/bin/bash` with effectively zero setup.
  • Python 3 is a standard dependency on Linux systems and widely available across agent development environments.
  • Neither requires a compile step, package manager bootstrap, or runtime installation dance just to execute a small helper script.
Discouraged runtimes
  • Node.js scripts (`.js`, `.ts`) require a separate runtime install and are not guaranteed to exist in every skill environment.
  • Go files (`.go`) require the Go toolchain and do not run directly as portable scripts.
  • Adding these runtimes increases setup friction, broadens dependency surface area, and reduces cross-runtime portability.
  • Our skill evaluator treats these non-portable script types as soft portability warnings so agents and maintainers can see the tradeoff clearly.
