Positioning

Why agent skills over slash commands?

Slash commands are useful shortcuts, but they are a weak foundation for portable, evaluable agent capabilities. Skills can encode activation context, inputs, outputs, examples, and reusable behavior. That makes them much easier to inspect and compare.

Question

Why not just score Claude Code slash commands?

  • Commands are imperative one-shot triggers, not portable capability definitions.
  • They are usually coupled to one runtime's invocation model.
  • They rarely explain when they should trigger, what inputs they require, or what outputs they return.
  • That makes consistent evaluation much harder.
Question

Why evaluate the IO spec instead of one vendor's runtime?

  • The evaluator focuses on the portable skill definition itself.
  • Trigger definitions, input contracts, outputs, and examples matter more than runtime-specific syntax.
  • The goal is vendor-neutral reuse across Claude Code, Codex, Gemini CLI, Cursor, and similar environments.
Question

Why prefer .sh and .py helper scripts?

  • They are the most portable script options across Linux and macOS environments.
  • They usually do not require a runtime manager, compile step, or package bootstrap.
  • Runtime-heavy script formats increase friction and reduce portability.
Question

When does an evaluation leave traces and when does it not?

  • Deterministic-only runs do not send uploaded skill content to third-party AI providers.
  • The package is still processed locally for the duration of the run.
  • After the completed result is delivered to the browser, transient server-side progress and result state is released.
  • The final JSON remains available in the current browser tab for viewing and download, but it is not kept after refresh.
  • If optional AI review is enabled, uploaded skill content for that run is sent to the single provider selected for that run after deterministic evaluation.
Question

Is there a prebuilt Docker image?

  • Yes. A prebuilt application image is published as ghcr.io/sasa-fajkovic/agents-skill-eval-app:latest.
  • It bundles the Go backend, frontend assets, Redis, and deterministic evaluator into a single deployable container.
  • For local development, you can still build the image directly from docker/Dockerfile.app.
Question

Is the project open source and self-hostable?

Yes. The repository is public and the frontend ships as raw HTML, CSS, and JavaScript served by a Go backend. That keeps the deployment path simple and transparent.

Evaluation model

Why these checks exist.

Every check targets a concrete risk: wasted tokens from loose triggers, broken portability from vendor-locked helpers, or security gaps that surface only after a skill is already deployed across teams.

Customization

Tune scoring to match your team's standards.

All scoring parameters live in a single file: backend/config.json. Adjust deduction points, caps, tier boundaries, and the weight split between deterministic and AI review scores. Changes take effect on the next server restart.

backend/config.json — scoring
{ "scoring": { "error_penalty": 5, // points deducted per ERROR finding "error_cap": 70, // max total deduction from errors "warning_penalty": 2, // points deducted per WARNING finding "warning_cap": 24, // max total deduction from warnings "deterministic_weight": 0.6, // weight of deterministic score in final blend "llm_weight": 0.4, // weight of AI review score in final blend "tiers": [ { "name": "excellent", "min_score": 90 }, { "name": "very_good", "min_score": 80 }, { "name": "good", "min_score": 70 }, { "name": "needs_work", "min_score": 50 }, { "name": "poor", "min_score": 0 } ] } }
  • Deterministic score starts at 100 and subtracts error_penalty per error (capped at error_cap) and warning_penalty per warning (capped at warning_cap).
  • AI review blend only applies when optional AI review is enabled. The final score is deterministic_weight × det_score + llm_weight × llm_score. Without AI review, the deterministic score is used directly.
  • Tiers are evaluated top-to-bottom; the first tier whose min_score the final score meets or exceeds becomes the quality tier.
